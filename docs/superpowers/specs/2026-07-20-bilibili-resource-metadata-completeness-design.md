# Bilibili 批量资源与直播元数据补全设计

## 背景

资源解析器使用 yt-dlp 的扁平播放列表预览来识别资源种类和限制任务规模。
这种模式对部分资源只返回 URL 或 ID：

- 番剧 `ep` 链接只返回当前一集，未展开所属季度。
- 番剧季度和收藏夹条目可能缺少名称、时长与封面。
- 直播元数据不包含主播名称。
- Bilibili 图片 CDN 会拒绝携带本地或 Tauri `Referer` 的封面请求，
  导致可信封面在桌面界面显示为破图。

## 目标

- 番剧 `ep` 和 `ss` 链接均列出所属季度的剧集；仅当季度超过现有 100 项
  单任务硬上限时截断并明确提示。
- 剧集条目展示真实名称、时长和可信封面。
- 收藏夹保持最多 100 条预览，并为这些条目展示真实名称、时长和可信封面。
- 直播展示真实主播名称；补充接口失败时不阻断直播解析。
- 所有封面图片不发送 `Referer`，同时保持现有严格 CSP 和可信域名校验。
- 所有新增 Bilibili API 请求使用统一的真实桌面浏览器格式请求头。

## 解析策略

### 番剧

首次解析继续用于识别资源种类。若输入为单集 `ep`，只接受 yt-dlp
元数据中的纯数字 `season_id`，并构造精确的
`https://www.bilibili.com/bangumi/play/ss<season_id>` 地址。季度地址随后使用
非扁平模式完整解析。

若输入已经是季度 `ss`，直接对原地址执行完整解析。结果仍服从应用现有的
单任务 100 项安全上限；超过上限时明确标记截断。

### 收藏夹

首次扁平解析负责识别收藏夹并获得总数。随后对同一规范化收藏夹地址执行
非扁平解析，使用 `playlistend=101` 判断是否超过上限，并只向前端返回前
100 条完整条目。每个条目从完整结果提取名称、时长、可信封面和规范化 URL。

完整解析可能比当前预览慢，但不会下载媒体文件。

### 直播

yt-dlp 继续负责直播可用性、标题、封面和录制信息。若结果缺少主播名称：

1. 请求 Bilibili 房间信息接口，只接受成功响应中的正整数 UID。
2. 使用 UID 请求主播信息接口，只接受长度受限的非空名称。
3. 任一请求失败、被风控、超时或返回无效结构时，保留 yt-dlp 结果并将主播
   留空，不让元数据补充失败阻断直播解析。

响应使用严格 Pydantic 模型验证，限制响应大小，不记录查询参数、Cookie
或完整响应。

## 请求头

新增房间元数据请求复用统一请求头常量：

- Windows 10 x64 Chrome 格式的真实桌面浏览器 `User-Agent`。
- 精确的 Bilibili HTTPS `Referer`。

UA 使用固定、可测试的浏览器格式，不随机生成版本，也不伪装移动端。QR
登录等需要明确应用标识的协议请求不改用浏览器 UA。

## 封面加载

`VideoPreview` 和 `ResourcePreview` 的远程封面 `<img>` 设置
`referrerPolicy="no-referrer"`。Bilibili CDN 因此不会收到本地端口或
Tauri 来源作为 `Referer`，可正常返回图片。

后端仍通过 `safe_cover_preview_url()` 只允许受信任的 Bilibili 图片域名并
升级为 HTTPS；不新增开放图片代理，也不放宽 CSP。

## 测试

- `ep` 元数据中的合法 `season_id` 会触发季度完整解析并列出多集。
- 非数字或缺失 `season_id` 不会用于构造地址，并安全回退当前单集。
- `ss` 地址直接完整解析，剧集包含真实名称、时长和可信封面。
- 收藏夹完整解析只返回前 100 条，正确设置总数和截断警告。
- 互动视频与普通视频既有两阶段行为不回归。
- 直播房间与主播接口成功时填充名称；错误码、无效 JSON、超时和无效 UID
  均安全回退。
- 两个封面组件都渲染 `referrerpolicy="no-referrer"`。
- 运行 Python 严格类型检查、完整后端测试、前端类型检查、组件测试、
  Playwright E2E、生产构建和桌面 Rust 检查。

## 非目标

- 不突破现有每个批量任务 100 项的上限。
- 不绕过会员、付费、地区、账号或 DRM 限制。
- 不为封面建立通用远程代理。
- 不修改 QR 登录请求头或第三方构建下载请求头。
