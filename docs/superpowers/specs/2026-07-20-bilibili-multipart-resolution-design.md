# Bilibili 普通分 P 视频解析修复设计

## 背景

资源解析接口当前统一启用 yt-dlp 的 `extract_flat: "in_playlist"`。
普通分 P 视频因此只返回缺少 ID、标题、作者、时长和格式信息的 URL
占位条目。后端将这些条目当作批量资源预览，前端只能显示
`item-1`、`item-2` 等占位内容。

## 目标

- 普通单 P 和多 P 视频进入现有视频详情流程。
- 多 P 视频展示真实分 P 标题、UP 主、时长和可用清晰度。
- 视频详情展示 yt-dlp 返回的真实封面，并继续使用现有受信任域名校验。
- 互动视频继续显示分支资源列表。
- 收藏夹、课程、番剧等批量资源继续使用有界扁平预览。
- 不输出 Cookie、签名媒体 URL 或其他敏感解析数据。

## 设计

资源解析采用两阶段策略：

1. 首次解析保持 `extract_flat: "in_playlist"`，用于低成本识别资源种类
   并生成批量资源预览。
2. 若首次结果被识别为普通 `VIDEO`，使用相同认证配置再次执行完整元数据
   解析，不启用 `extract_flat`。
3. 将完整结果交给现有 `resolved_video_from_info()`，生成
   `ResolvedVideo`。该模型同时通过 `safe_cover_preview_url()` 提取并校验
   根视频或首个分 P 的封面，由前端进入现有视频详情和分 P 下载流程。
4. 若首次结果被识别为 `INTERACTIVE` 或其他资源种类，不执行第二阶段，
   保留当前资源列表行为。

第二阶段沿用现有 Cookie 上下文、基础 yt-dlp 选项和错误映射。解析请求仍为
只读元数据请求，不下载媒体文件。

## 错误处理

- 任一解析阶段的 Cookie 解密错误继续映射为现有
  `cookie_decryption_failed`。
- yt-dlp 和网络错误继续通过现有 `map_engine_error()` 脱敏。
- 完整解析返回无效元数据时继续使用现有 `invalid_metadata` 错误。
- 不在日志或错误响应中加入解析结果、媒体 URL、Cookie 或认证信息。

## 测试

- 普通多 P 视频：首次扁平解析后执行完整解析，返回真实视频详情与分 P。
- 普通单 P 视频：执行完整解析并返回现有视频详情结构。
- 视频封面：可信 Bilibili 封面地址被保留并升级为 HTTPS，非可信地址被丢弃；
  前端视频详情继续渲染封面或安全占位内容。
- 互动视频：只执行扁平解析，保留分支列表和警告。
- 收藏夹：只执行扁平解析，继续限制预览数量。
- 验证两次解析使用相同认证选项，第二次不包含 `extract_flat`。
- 运行 Python 测试与严格类型检查，并运行前端类型检查和相关测试，
  确认 API 形状和现有渲染流程未回归。

## 非目标

- 不修改前端视觉设计。
- 不改变互动视频分支下载能力的现有限制。
- 不调整其他资源类型的下载策略。
- 不新增依赖。
